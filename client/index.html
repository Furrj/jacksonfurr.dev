Reflect on the process of enhancing the artifact.

    I mainly learned how the functionality of a language naturally influences the architecture and logical flow of the software written with it. For example, Express and JS tend to lend themselves to more of a straightforward, simple routing pattern when it comes to writing HTTP server functionality. Enhancement 3: Databases

Briefly describe the artifact. What is it? When was it created?

    This artifact is the server for a personal project called “Todo 2.0”. It is a web application that allows users to add and remove “todo” items from a list, so that they may track their productivity. It runs on the MERN stack currently, meaning it uses JS as the core language, Node as the JS runtime, Express as the HTTP server, MongoDB as the database, and React for the frontend client. It was initially created a little over 2 years ago as a personal project so that I could better learn how the MERN stack functioned.


Justify the inclusion of the artifact in your ePortfolio.

    I selected this item because it presented a great chance to show my skills regarding implementing a database into a software project. This program needs a database so that the Todo items can be used across different sessions, as storing them in-memory would mean they would disappear whenever the executable stops running, making updates impossible. At the same time, it only needed two database objects, the Todo and User objects, which means it was fairly straightforward to implement. I implemented communication with a Postgres backend which consisted of a database named 'Todo' and two tables, one named “Users” and one named “Todos”. The API routes interacted with this database through a driver for Golang known as 'pgx' and I used prepared SQL queries to interact with Postgres directly instead of using an ORM.


Reflect on the process of enhancing the artifact.

    I learned that, while good practice for SQL, writing prepared statements instead of using an ORM can make the process more difficult, especially when it comes to refactors. Anytime I wanted to add a column to a database table, it required making changes in three separate places: the SQL statements for creating the table, the SQL statements for querying the table, and the Go functions for executing the SQL statements. Although ORM's can slow down performance, in bigger more complicated projects I can see how they may be useful. A good compromise would be using a code-gen solution that allows you to still use SQL but generates the statements for you based on the Golang structs.
